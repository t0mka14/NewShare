Clinical Speech Recording Desktop App - Design Requirements Specification

================================================================================
1. Overview
================================================================================

The application is a clinical desktop tool for recording speech data from
participants. It records a single continuous master WAV file per session,
allowing clinicians to mark task boundaries during recording. After the session,
the examiner optionally refines boundaries in a built-in waveform editor. Final
clips are exported from the unchanged master recording, archived as a ZIP, and
uploaded to a REST API server.

A companion auto-updater native executable handles version checking and
updating the application before launch.

================================================================================
2. Repository and Project Structure
================================================================================

The application and the auto-updater live in a single monorepo managed by
Gradle with multiple subprojects.

Monorepo layout:

  <repo-root>/
    app/                        <- main clinical recording application
    updater/                    <- auto-updater native executable
    shared/                     <- shared Kotlin code (version model, API
                                   contract, common utilities)
    buildSrc/                   <- shared build logic and Gradle conventions
    gradle/
      libs.versions.toml        <- centralized version catalog for all modules
    settings.gradle.kts         <- includes all subprojects
    build.gradle.kts            <- root-level build configuration

Gradle subproject responsibilities:

  app/
    - Main Compose Multiplatform Desktop application.
    - Depends on :shared for version model and update check contract.
    - No dependency on :updater.

  updater/
    - Auto-updater compiled to a native executable using Kotlin/Native or
      GraalVM native-image.
    - Responsible only for: version check, download, file replacement,
      launching the app.
    - Depends on :shared for version model and update check contract.
    - No dependency on :app.

  shared/
    - Lightweight Kotlin module.
    - Contains: AppVersion model, VersionCheckResponse model, Ktor-based
      VersionCheckClient, common utilities.
    - No dependency on :app or :updater.

  buildSrc/
    - Shared Gradle build conventions (Kotlin JVM setup, dependency versions,
      common plugin configuration).

The centralized version catalog (libs.versions.toml) manages all dependency
versions across all subprojects to avoid duplication and version drift.

================================================================================
3. Auto-Updater
================================================================================

The auto-updater is a native executable compiled from Kotlin code. It is the
entry point that users launch. It updates the app if needed and then starts it.

Build approach:
- Compile to a native executable using GraalVM native-image or Kotlin/Native.
- This eliminates the need for a separate JVM installation for the updater
  itself.
- The updater uses the app's bundled JRE to launch the main application.

Updater responsibilities:
1. Check the version endpoint on the server for the latest available version.
2. Compare with the currently installed app version.
3. If a newer version is available:
   a. Download the update package from the server.
   b. Verify the package checksum (SHA-256 or similar).
   c. Back up the current app/ directory.
   d. Extract/replace the app/ directory contents.
   e. On failure: restore the backed-up app/ directory.
4. Launch the main application using the bundled JRE.
5. Exit.

The updater must NOT be located inside the app/ directory it replaces during
updates.

Installation directory layout:

  <install_dir>/
    updater.exe                  <- native updater executable (entry point)
    runtime/                     <- shared bundled JRE (jlink image)
      bin/
        java (or java.exe)
      ...
    app/                         <- main application (replaced during update)
      app.jar                    (or equivalent Compose Desktop output)
      resources/
        strings_en.xml
        strings_cs.xml
        ...
      version.json               <- current installed app version
    data/                        <- application data (never touched by updater)
      config/
      sessions/
      upload_queue/
      logs/

The updater launches the app as:
  <install_dir>/runtime/bin/java -jar <install_dir>/app/app.jar

Shared JRE:
- One jlink-built JRE image is shared by both the updater (for launching app)
  and the app itself.
- This avoids duplicating a 60-120 MB runtime.
- JRE updates are versioned separately and only applied when explicitly
  included in an update package.

Updater version check API contract (defined in :shared):
- GET /api/version/latest
- Response: { "version": "1.2.3", "downloadUrl": "...", "checksum": "..." }
- AppVersion and VersionCheckResponse models are in :shared.

Update package:
- A ZIP archive containing updated app/ directory contents.
- Does NOT contain the runtime/ or data/ directories.
- Contains version.json with new version identifier.

Rollback:
- Before replacing app/ contents, the updater creates a backup of the current
  app/ directory.
- If the new app fails to launch or the update fails mid-way, the backup is
  restored on next updater run.

Silent/offline behavior:
- If the version check endpoint is unreachable, the updater proceeds to launch
  the existing app without updating.
- Network errors during update are logged and trigger rollback.

5. Architectural Principles
================================================================================

Use a clean, layered, testable architecture in the :app module:

  Compose UI
    -> Decompose Components / ViewModels
    -> Use Cases
    -> Repository / Service Interfaces
    -> Implementations (Filesystem, Audio, Network, Clock, ID Generation)

Rules:
- UI contains no business logic.
- Business rules live in use cases and domain services.
- All external systems (filesystem, microphone, network, clock) are abstracted
  behind interfaces.
- Interfaces enable fake/mock implementations for testing without real hardware
  or network.

9. Participant Input
================================================================================

Before the examination, the examiner enters participant and clinical metadata.

Fields examples:
- Patient Code (used in folder names and filenames)
- Sex
- Birth Year
- Investigator Initials
- Visit Number (validated with regex, e.g., "V\d+")
- Stimulation State (ON / OFF)
- Medication State (ON / OFF)
- Project Name
- Notes (free text, optional)

Requirements:
- The fields required can be configured in the settings.
- Store the current setup in a setting Json file.
- Validate required fields before allowing continuation.
- Field validation supports regex-based rules configurable per field.
- Use only the Patient Code in filenames and folder names, not full names.
- On confirmation, generate a session ID and create the session folder on disk.
- Store participant metadata as participant.json in the session folder.

================================================================================
10. Calibration Screen
================================================================================

The calibration screen is mandatory and appears between participant input and
the examination.

Purpose:
- Allow the examiner to verify the microphone is correctly connected and the
  signal level is appropriate before recording begins.

Behavior:
- Show live audio input level from the selected microphone.
- Display a bar graph showing the observed level and also recommended level target.
- The examiner confirms calibration is acceptable before proceeding.
- Calibration result (device name, observed level range) may be stored in
  examination metadata for audit purposes.
- If the microphone is unavailable, show a clear error with guidance.

This screen does not start the master recording. Recording begins only after
calibration is confirmed and the examination starts.

================================================================================
12. Continuous Recording Model
================================================================================

The application records one continuous master WAV file for the entire
examination session.

Key rules:
- Recording starts after calibration is confirmed and the examination begins.
- Recording continues uninterrupted through all task screens.
- Task Start and Stop buttons do not start/stop the microphone. They create
  timestamped events marking task boundaries within the continuous recording.
- Recording stops when the examiner ends the examination.
- The master recording is never modified after creation.
- Writing uses a temporary file first, finalized and renamed on completion.

Audio format (default):
- Container: WAV
- Encoding: PCM signed
- Sample rate: 48000 Hz
- Bit depth: 16-bit
- Channels: mono (1)

Interfaces:
- ContinuousSessionRecorder
- AudioInputDeviceProvider
- AudioLevelMeter (exposes RMS/peak values for UI feedback)
- AudioPlaybackService (for example audio and editor playback)
- AudioFormatProvider

================================================================================
13. Task Event Timeline
================================================================================

During recording, the app records a timeline of timestamped events.

Event types:
- SESSION_RECORDING_STARTED
- TASK_SCREEN_ENTERED
- REPETITION_STARTED
- START_BUTTON_PRESSED         (marks recording start for this repetition)
- STOP_BUTTON_PRESSED          (marks recording end for this repetition)
- REPETITION_ACCEPTED
- REPETITION_REJECTED          (when examiner repeats)
- TASK_COMPLETED
- SESSION_RECORDING_STOPPED

Each event stores:
- event type
- absolute clock timestamp
- offset in milliseconds from the start of the master recording
- task id
- repetition number
- attempt number

Timestamps relative to the master recording (millisecond offsets) are the
primary reference for audio cutting.

Repeat attempts:
- Rejected attempts are preserved in metadata for audit purposes.
- Only the accepted attempt is used for final clip export.
- Attempt numbering is preserved.

Domain models:
- RecordingSession
- TaskTimeline
- TaskEvent
- TaskSegment
- TaskRepetitionAttempt (accepted / rejected)

Timeline is saved to:
  <session>/timeline_original.json

If manual cleaning is used, edited boundaries are saved to:
  <session>/timeline_edited.json

The original timeline is never modified.

================================================================================
14. Examination / Task Recording Screen
================================================================================

The clinician is guided through configured task screens during the examination.

Each task screen shows:
- Task name (from localization resources).
- Task instruction text (from localization resources, rendered as rich text).
- Optional: button or auto-play for example audio (e.g., aaa.wav).
- Current task index and total task count.
- Current repetition and total repetitions.
- Start button (creates START_BUTTON_PRESSED event).
- Stop button (creates STOP_BUTTON_PRESSED event).
- Repeat/Again button (marks current attempt as rejected, allows new attempt).
- Next/Continue button (enabled after a valid accepted segment exists).
- Cancel examination option (with confirmation dialog).
- Live recording feedback indicator (based on settings).

Button state rules:
  Idle:       Start=enabled, Stop=disabled, Repeat=disabled, Next=disabled
  Recording:  Start=disabled, Stop=enabled, Repeat=disabled, Next=disabled
  Stopped:    Start=disabled, Stop=disabled, Repeat=enabled, Next=enabled
  Failed:     Start=enabled, Stop=disabled, Repeat=disabled, Next=disabled

================================================================================
15. Live Recording Feedback
================================================================================

During examination task screens, show real-time microphone signal feedback.

Configurable modes (from settings):
- Pulsating circle: radius/opacity pulsates based on current RMS amplitude.
- Live amplitude envelope: rolling window canvas of recent amplitude values.
- Off.

Implementation rules:
- Metering is derived from the same audio stream being written to disk.
- Must not interfere with disk writing.
- Use RMS or peak amplitude, not raw samples.
- Envelope buffer holds a short rolling window (e.g., last 2-3 seconds).
- Update rate should be sufficient for smooth animation (e.g., ~30 fps or
  tied to audio buffer callbacks).

Relevant components:
- AudioLevelMeter
- AmplitudeEnvelopeBuffer
- PulsatingAmplitudeIndicator (Compose component)
- LiveAmplitudeEnvelopeView (Compose Canvas component)

================================================================================
16. Manual Fine Cleaning / Audio Editor
================================================================================

If manual fine cleaning is enabled in settings, this screen is shown after the
examination ends and before the processing screen.

The editor shows one task/repetition segment at a time.

Waveform display:
- Loads the master recording.
- Displays waveform for the region:
    (start_timestamp - X seconds) to (stop_timestamp + X seconds)
  where X is a constant defined in code.
- Waveform is rendered as downsampled min/max peak pairs (not raw samples).
- Downsampled data is cached per session in a waveform cache file.
- UI renders only the visible portion of the waveform.

Vertical boundary lines:
- Start boundary line (movable, initially at START_BUTTON_PRESSED offset).
- Stop boundary line (movable, initially at STOP_BUTTON_PRESSED offset).
- Playback position line (moving, shown during playback).

Interactions:
- Drag start/stop boundary lines to adjust clip boundaries.
- Play button: plays audio between current start and stop boundary lines.
- Stop playback button.
- Accept/confirm button: saves adjusted boundaries and moves to next segment.
- Previous/next segment navigation.

On confirmation:
- Adjusted boundaries for all segments are saved to timeline_edited.json.
- The master recording file is not modified.

Waveform performance requirements:
- Generate waveform overview data using min/max peak downsampling.
- Cache waveform data to a file per session (e.g., master_waveform.cache).
- Regenerate cache if missing.
- Do not render all raw samples. Render only what fits the visible canvas
  width.

Relevant models and services:
- ManualCleaningSettings
- WaveformOverviewGenerator
- WaveformCacheRepository
- TaskClipEditModel
- BoundaryAdjustmentState
- EditedTimelineRepository

================================================================================
17. Processing Screen
================================================================================

Shown after examination (and after manual cleaning if enabled) while final
clips are being produced.

Behavior:
- Show progress indicator and status message.
- Prevent accidental navigation or window closure during processing.
- Handle cancellation safely if the user requests it.
- On completion, proceed to summary.

Processing steps:
1. Determine boundaries: use timeline_edited.json if it exists, otherwise
   timeline_original.json.
2. Extract each accepted task/repetition segment from the master WAV.
3. Write exported clips to the session clips/ folder.
   Filename format: PatientCode_TaskName_RepN.wav
4. Zip the entire session folder to the session archive/ folder.
5. Update examination.json with clip paths and processing status.

Relevant services:
- ProcessRecordingSessionUseCase
- AudioSegmentCutter
- ClipExporter
- ZipArchiver
- PostRecordingProcessor

================================================================================
18. Summary and Upload
================================================================================

After processing, show the examination summary.

Summary screen:
- List all tasks and their exported clips.
- Show any missing or failed clips.
- Show upload status.
- Allow manual upload trigger.
- Allow return to editor if re-processing is needed.

Upload behavior:
- Use HTTPS.
- Use Ktor Client with multipart support.
- Include secret/token-based authentication as required by the server API.
- Upload package includes:
    - participant.json
    - examination.json
    - task_configuration_snapshot.json
    - timeline_original.json
    - timeline_edited.json (if present)
    - exported task clips (WAV files)
    - session ZIP archive
    - optionally the master recording if required by the backend
- Do not delete local data after upload.
- Store upload result and server-assigned IDs in upload_status.json.
- Mark statuses: NotUploaded, Uploading, Uploaded, Failed, PartiallyUploaded.
- Failed uploads remain in upload_queue/queue.json for retry.
- Retry logic should handle network interruptions gracefully.

Relevant services:
- UploadRepository
- UploadQueueRepository
- UploadExaminationUseCase
- RetryFailedUploadsUseCase

20. Localization and Rich Text
================================================================================

All user-facing strings are loaded from external XML files.
No string literals may appear in UI code.

String resource files:
- Located in the app resources folder: <install_dir>/app/resources/
- Named by locale, e.g.: strings_en.xml, strings_cs.xml, strings_de.xml
- Language selected based on system locale or explicit setting.

Rich text markup:
- String values may contain inline markup tags:
    <bold>text</bold>
    <italic>text</italic>
- Combinations result in bold-italic rendering if needed.
- The localization layer parses these tags and produces a structured RichText
  model, not raw XML strings.
- Compose renders RichText as AnnotatedString.
- Legacy XML-style tags (_b, /b, etc.) are NOT supported.

Example XML entry:
  <string key="task.phonation.instruction">
    Please say <bold>aaaa</bold> for as long as possible.
  </string>

Relevant components:
- LocalizedStringProvider
- XmlStringResourceLoader
- RichTextParser
- RichText (domain model: list of Segment(text, style))
- RichTextRenderer (Compose)

26. Clinical Safety and Data Integrity
================================================================================

- The original master recording is never modified after creation.
- The original timeline is never modified. Edits go to timeline_edited.json.
- Local data is never deleted after upload.
- Each session can always be reprocessed from master + timeline.
- Use patient codes in filenames, not full personal identifiers.
- Use HTTPS for all uploads.
- Store server response and IDs locally after upload.
- Consider storing file checksums (e.g., SHA-256) for integrity verification.
- Store enough metadata to fully audit how each clip was produced.
- Each examination stores a snapshot of the task configuration at time of
  recording.
- Log all recording start/stop events, processing steps, and upload outcomes.
- The updater must never modify the data/ directory.
- The updater must always maintain a rollback backup before applying updates.
- The app version is stored in version.json inside app/ and is read by the
  updater to determine whether an update is needed.

27. Key Design Summary
================================================================================

The project is a Gradle monorepo containing three subprojects: :app, :updater,
and :shared. The auto-updater is a native executable (GraalVM or Kotlin/Native)
that users launch directly. It checks a version endpoint, optionally updates
the app/ directory, and then starts the main application using the shared
bundled JRE. The updater and app share one jlink-built JRE image to avoid
duplicating the runtime.

The main application records one continuous master WAV file for the entire
examination. A mandatory calibration screen precedes each examination to verify
microphone levels. Task Start/Stop buttons create timestamped markers within
the continuous recording. After examination, the app either automatically cuts
clips from those markers or, if manual fine cleaning is enabled, presents a
waveform-based boundary editor where the examiner can refine clip start/stop
points. Final clips are exported from the unchanged master recording, then the
session is archived as a ZIP. All metadata, configurations, timelines, and
upload status are stored as JSON. Data lives in a "data" subfolder inside the
application installation directory and is never touched by the updater.

The app uses Compose Desktop for UI, Decompose for navigation and components,
Ktor Client for uploads, and a layered clean architecture with testable
interfaces throughout.
